# Proposal Admission Model

## 1. Purpose

This document defines a generic admission lifecycle for governed proposals.

The model exists to ensure that governed objects may be:

- proposed
- procedurally validated
- explicitly adopted
- sealed
- made activation-eligible

without allowing proposal generation or procedural validation to become authority.

This document is a protocol extension.
It is not part of the minimal Boundary Protocol core.

## 2. Core Principle

```text
Proposal is not authority.
Validation is not adoption.
Adoption is not activation.
```

Each step is distinct and MUST be represented explicitly where this extension is used.

## 3. Governed Object Lifecycle

A governed object enters the system through the following lifecycle:

```text
proposal
→ procedural validation
→ adoption
→ sealing
→ activation eligibility
```

This lifecycle applies only to governed object classes that explicitly adopt this extension.

## 4. Lifecycle Stages

### 4.1 Proposal

A proposal is a candidate object submitted for possible admission.

A proposal:

- MAY be generated by an agent
- MUST be self-contained
- MUST be structurally inspectable
- MUST NOT be treated as authoritative

### 4.2 Procedural Validation

Procedural validation checks only whether the proposal satisfies required structural and governance constraints.

Procedural validation MAY include:

- schema completeness
- required field presence
- profile consistency
- identifier validity
- boundary compliance

Procedural validation MUST NOT include:

- market judgment
- semantic worth judgment
- utility judgment
- activation decision
- adoption decision

### 4.3 Adoption

Adoption is the explicit decision to accept a procedurally valid proposal as a governed object.

Adoption:

- MUST be explicit
- MUST be represented by an adoption artifact
- MUST be authority-bearing
- MUST be distinct from validation

### 4.4 Sealing

Sealing binds the admitted object to specific artifacts and versions.

Sealing MAY include:

- hashes
- manifests
- version identifiers
- supporting evidence bundles

A sealed object becomes reproducible and auditable.

### 4.5 Activation Eligibility

Activation determines whether the sealed object may be used by downstream runtime or governance processes.

Activation eligibility:

- MUST NOT be implied by proposal existence
- MUST NOT be implied by procedural validation alone
- MUST NOT be implied by adoption alone unless explicitly defined

## 5. Authority Boundary

The admission model enforces the following separation:

| Layer | Role |
|------|------|
| Proposal layer | Generates candidate objects |
| Validation layer | Checks procedural compliance |
| Adoption layer | Grants authority |
| Sealing layer | Produces immutable evidence |
| Activation layer | Permits downstream use |

This separation is mandatory wherever this extension is applied.

## 6. Generic Admission Object

The admission model does not assume a specific domain.

A governed object MAY be:

- a profile
- a checklist family
- a policy pack
- an engine capability profile
- a structural observation model
- another explicitly governed object class

This model governs lifecycle only.
It does not define domain semantics.

## 7. Required Artifacts

A governed proposal SHOULD have the following artifact classes:

- proposal artifact
- supporting description artifact
- prototype or realizability artifact
- adoption artifact
- sealing artifact

Exact file names MAY vary by domain.

## 8. Procedural Validation Scope

Procedural validation is intentionally narrow.

It MAY validate:

- schema validity
- identifier uniqueness
- structural consistency
- declared profile completeness
- bounded metadata correctness

It MUST NOT validate:

- business value
- legal sufficiency
- semantic correctness
- usefulness
- intent

Those are outside the authority of procedural validation.

## 9. Fail-Closed Policy

The model operates in fail-closed mode.

A proposal MUST NOT advance if:

- required artifacts are missing
- structural validation fails
- adoption artifact is absent
- sealing requirements are incomplete
- activation boundary is unspecified where required

No object becomes runtime-eligible through partial success.
